This is the vEcoli whole-cell modeling project. For years they have only ingested a single rnaseq dataset into the parca (parameter calculator - the thing that reads all of the input data from ecocyc and otherwise, solves various parameters and stores them in sim_data in order to construct a running simulation that behaves (as much like as possible) E. coli the organism). Even this rnaseq source has adjustments made to certain critical genes (multiplying them by 10). Now there is an effort to pull in a number of other rnaseq datasets (currently in a sibling dir omics-sources) to feed into the parca as a source of various parameters. As we have tried this out a number of different kinds of failure occur. One is that one of the solvers in the parca fails to converge, and another is that there is an error in the sim itself: some counts go to zero. We want to debug this. The third kind of error is that the sim is "unhealthy" - no visible errors, but it doesn't grow well especially across generations. We want to find a way to explore these problems in a more systematic way - starting from our existing datasets from omics-sources. There are already adjustments to certain genes as noted before, the question is what is the relationship between input rnaseq datasets (and in the future other datasets) and the behavior of the simulation? What should it be? If we already have adjustments, can we understand alternate inputs through the adjustments required to get them to work? What should we reasonably expect in terms of the response of the simulation to different inputs? What makes some fail and others not? What makes some behavior reasonable and others not? As we suspect the problem is not just in the data or just in the model but some interaction between them, it also points to improvements in the simulation. What current design choices are holding us back from responding reasonably to different datasets? How do we find what about the current design is "fine-tuned" to specific conditions (ppGpp structure is suspicious, also the partitioning architecture needs to be replaced - what should take its place?) and what is "robust"? How does the actual operation of the cell compare to what our simulation is doing? I think the data ingestion question is at the nexus of these questions as it tests the range of the model, rather than reproducing certain already known conditions, and therefore shows us how the model breaks down. Can you please do a thorough analysis of the simulation in this repository and also look at the alterate input datasets - one thing we want to do also is design a better way to handle datasets like this rather than put them in the repo. Maybe something like DVC? How would we integrate that here and avoid putting these data files in the repo, and maybe moving out all the existing ones? Beyond that we want to do a kind of "sensitivity" analysis on the datasets. Starting with the one that works (vecoli) how can we systematically perturb it to build a model of the space of input data and how it relates to simulation behavior? What data is acceptable? How does it change as we move? In order to do this we want to use the newly created multi-parca-workflow approach - can we create a large workflow that most effectively searches through the space of data in order to assemble a picture of simulation behavior? and also to guide us in how to improve it? Let's say we can run 10,000 individual simulations, what is the best way to organize the work to get the most information from our input data sets and the perturbations provided by some recent work by Annabelle (I can't find it right now) that does different operations to datasets to get datasets in some proximity to the original given a certain relationship (the simplest would be to add some noise to everything) - we can develop more of these as well. In order to do this we need to generate a number of "synthetic" datasets from our existing ones and store them in some kind of systematic way (maybe using DVC again?) Organizing inputs and results will be a special challenge of this project. Can we develop a plan along these lines and save it in .claude/plans/ ?

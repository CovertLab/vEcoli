## Goal

Support running multiple ParCa instances with different RNAseq datasets (or other
differing `parca_options`) in a single Nextflow workflow, feeding independent
variant+simulation chains that are still comparable via the existing multi-variant
analysis layer.

---

## Mental Model

Two-level indexing:

```
parca_idx  (0, 1, 2 ...)       ← which ParCa / RNAseq dataset
  └─ variant_idx (0, 1, 2 ...) ← which sim_data variant within that ParCa
```

Global variant index = `parca_idx * pickles_per_parca + variant_idx`

This keeps every `.cPickle` filename unique and every simulation output directory
non-colliding, while the existing `analysisMultiVariant` layer naturally compares
across all of them.

`pickles_per_parca` = `n_variants + (1 if not skip_baseline else 0)` and is
computed at workflow-generation time by `_count_pickles_per_parca()` in
`workflow.py`.

## Implementation: What Was Actually Built

### Actual vs Planned differences

The original plan used Nextflow process *aliasing* (`include { runParca as runParca_0 }`).
The actual implementation instead passes a **channel of tuples** to a single
`runParca` process, letting Nextflow fan out naturally. This is simpler and avoids
the need for alias imports.

The `createVariants` call (previously hardcoded in `template.nf`'s workflow block)
was moved into the dynamically generated `RUN_PARCA` block so `generate_code()`
has full control over both ParCa and variant setup.

A new `mergeVariantMetadata` process handles combining the per-parca
`metadata_{parca_id}.json` files into the single `metadata.json` that all
downstream analysis processes expect.

## File-by-File Changes (as implemented)

### 1. `configs/default.json` ✓
Added `"parca_variants": []` top-level key.

**Usage**: each entry is a dict of `parca_options` overrides merged on top of the
baseline `parca_options`. Empty list → single ParCa run (fully backward-compatible).

```json
{
  "parca_options": {
    "rnaseq_manifest_path": "reconstruction/ecoli/experimental_data/rnaseq/manifest.tsv",
    "rnaseq_basal_dataset_id": "ref_0001"
  },
  "parca_variants": [
    { "rnaseq_basal_dataset_id": "ref_0001" },
    { "rnaseq_basal_dataset_id": "gbw_0001_v2" }
  ]
}
```

See `configs/test_multi_parca.json` for a complete working example.

---

### 2. `runscripts/create_variants.py` ✓

**`apply_and_save_variants()`**: new `offset: int = 0` parameter.
- Baseline metadata key shifts to `offset` (was always `0`)
- Variant pickle `i` is saved as `{offset + i + 1}.cPickle`
- Metadata keys are `{offset}` through `{offset + n_variants}`

**`main()`**: new `--offset INT` CLI argument (default 0). Reads from config as
`config["offset"]`. Baseline pickle filename and no-variant metadata key both
respect the offset.

Backward-compatible: offset=0 produces identical output to the old behavior.

---

### 3. `runscripts/nextflow/template.nf` ✓

**`runParca` process**:
```nextflow
input:  tuple val(parca_id), path(config)
output: tuple val(parca_id), path('kb')
publishDir: parca_${parca_id}   // was: parca
```

**`createVariants` process**:
```nextflow
input:  path config, tuple val(parca_id), path(kb), val(offset)
output: path '*.cPickle'                        // variantSimData (unchanged)
        path "metadata_${parca_id}.json"        // variantMetadata (was: metadata.json)
script: adds --offset $offset; renames metadata.json → metadata_${parca_id}.json
```

**New `mergeVariantMetadata` process**:
- Input: `path 'metadata_*.json'` (Nextflow stages all collected files)
- Output: `path 'metadata.json'`
- Script: Python one-liner using `glob.glob('metadata_*.json')` to merge all dicts
- For single-parca workflows this is a trivial pass-through

**Workflow section**: the hardcoded `createVariants(params.config, kb) ...` block
was removed. Everything from `runParca` through `variantMetadataCh` is now
generated by `generate_code()` and substituted into `RUN_PARCA`.

---

### 4. `runscripts/workflow.py` ✓

**New `_count_pickles_per_parca(config)`**: dynamically loads `parse_variants`
from `create_variants.py` via `importlib` to count how many pickles each parca
run will produce. Used to compute non-overlapping offsets.

**`generate_code(config, parca_config_uris)`**: signature changed; now takes a
list of per-parca config file paths/URIs.

- Single-parca path (`parca_variants` absent/empty): generates
  `Channel.of([0, file('...')])` → identical Nextflow behavior to before
- Multi-parca path: generates `Channel.of([0, file(...)], [1, file(...)])` feeding
  a single `runParca` process (Nextflow fans out), then an offset channel,
  then `runParca.out.combine(parcaOffsetCh, by: 0)` to pair each kb with its offset
- `kb` for analysis = parca_0's kb (filtered by `parca_id == 0`)
- `sim_data_path` bypass: raises `RuntimeError` if combined with `parca_variants`

**`main()`**: before calling `generate_code()`, loops over `parca_variants_list`
to write `parca_config_{i}.json` files (merged `parca_options` + override) to
`local_outdir`, copies them to `outdir`, and collects their URIs.

## Known Limitations / Future Work

- **`kb` for analysis**: all analysis processes use parca_0's `validationData.cPickle`.
  For variants from parca_1, this is technically wrong but usually harmless
  (validation data doesn't depend on RNAseq). Proper fix: carry `kb` per variant
  through the sim channel.

- **`analysisParca`**: currently runs once using parca_0's kb. Should run once per
  `kb_i` to generate parca-level plots for each dataset.

- **Dataset labels in metadata**: metadata JSON records variant params but not
  which parca/dataset produced them. Analysis scripts that label axes by variant
  index won't know the dataset name without consulting the config. A future
  improvement would inject `parca_dataset_label` into each variant's metadata entry.

- **Cloud resumability**: per-parca config files are referenced by absolute path in
  the generated `.nf` file. For cloud runs they are copied to the output URI and
  that URI is used. This should work for gcloud/sherlock but has not been tested.

- **End-to-end test**: a Nextflow stub run with `test_multi_parca.json` has not
  been executed. That's the immediate next validation step.
